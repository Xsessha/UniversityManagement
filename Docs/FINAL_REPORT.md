# Фінальний звіт — Lumina University Management System

## Загальна інформація

**Назва проєкту:** Lumina University Management System  
**Технологічний стек:** ASP.NET Core 9, C#, Entity Framework Core, ASP.NET Identity, SignalR, xUnit  
**База даних:** SQLite (за замовчуванням), SQL Server (опційно)  
**Архітектурний стиль:** Clean Architecture, MVC

---

## Мета проєкту

Розробити веб-застосунок для управління університетом, що автоматизує ключові процеси: облік студентів і викладачів, курси, розклад, оцінювання, відвідуваність, звітність і сповіщення. Система замінює розрізнені таблиці та паперові журнали єдиною платформою з рольовим доступом.

---

## Реалізований функціонал

### Автентифікація та авторизація
Система використовує ASP.NET Core Identity з трьома ролями: Admin, Teacher, Student. Кожна роль має чітко визначені права доступу на рівні контролерів через атрибут `[Authorize(Roles = "...")]`. При першому запуску автоматично створюються тестові облікові записи для кожної ролі.

### Управління академічними даними
Реалізовано повний CRUD для: студентів, викладачів, курсів, груп, факультетів, занять, розкладу. Зв'язки між сутностями відображають реальну структуру університету: Факультет → Групи → Студенти, Викладач → Курси → Заняття.

### Оцінювання та відвідуваність
Викладачі виставляють оцінки студентам з прив'язкою до курсу. Відвідуваність відмічається для кожного заняття зі статусами: Present, Absent, Late. Студент бачить повну картину своєї успішності на сторінці деталей профілю.

### Звітність
Адміністратор генерує звіти у форматах PDF та Excel. Звіти зберігаються в архіві з датою та автором. Побудова звітів реалізована через патерн Builder, що дозволяє легко додавати нові формати.

### Сповіщення в реальному часі
Система сповіщень побудована на SignalR. Викладач надсилає повідомлення — студенти отримують їх без перезавантаження сторінки. Архітектурно реалізовано через патерн Observer, де `TeacherSubject` оповіщає підписників при кожній зміні.

### Dashboard
Головна панель агрегує ключові показники: кількість студентів, курсів, середній рейтинг, останні активності.

---

## Архітектурні рішення

### Clean Architecture
Система розбита на 5 проєктів із однонаправленими залежностями. Бізнес-логіка в `Services` не знає про EF Core чи HTTP — вона залежить лише від інтерфейсів `IRepository<T>`. Це дозволяє тестувати сервіси в повній ізоляції від бази даних.

### Патерни проєктування
У проєкті реалізовано 7 патернів GoF:

| Патерн    | Де застосовано                     | Проблема що вирішує                         |
|-----------|------------------------------------|---------------------------------------------|
| Factory   | Створення занять (Lecture/Seminar/Lab) | Інкапсуляція логіки створення об'єктів  |
| Strategy  | Розрахунок рейтингу студента       | Взаємозамінні алгоритми без зміни сервісу   |
| Observer  | Система сповіщень                  | Слабка зв'язаність між відправником та одержувачем |
| Builder   | Генерація PDF та Excel звітів (в процесі)     | Покрокова побудова складного об'єкта        |
| Composite | Ієрархія факультет→група→студент   | Однакова обробка листків і вузлів дерева    |
| Visitor   | Операції над ієрархією             | Додавання операцій без зміни доменних класів|
| Repository| Доступ до даних у сервісах         | Ізоляція бізнес-логіки від ORM              |

### Підтримка двох СУБД
Через конфігурацію `DatabaseProvider` система перемикається між SQLite (розробка) та SQL Server (production) без змін у коді. EF Core абстрагує різницю між провайдерами.

## Тестування

Написано 25 unit-тестів, всі проходять успішно.

| Файл тестів                    | Тести | Що перевіряється                              |
|--------------------------------|-------|-----------------------------------------------|
| `StudentTests.cs`              | 1     | Коректне створення моделі Student             |
| `StudentServiceTests.cs`       | 6     | GetAll, GetById, Add, Delete (+ edge cases)   |
| `StudentsControllerTests.cs`   | 5     | Index, Details (found/not found), Edit        |
| `FactoryTests.cs`              | 5     | Всі 3 фабрики + Provider + типи занять        |
| `AuthenticationServiceTests.cs`| 5     | Валідація паролів (Theory + edge cases)       |
| `RoleAccessTests.cs`           | 1     | Рольова модель                                |

**Підходи:** Mock (`Moq`) для ізоляції сервісів, InMemory Database для контролерів, `xUnit` як фреймворк, `FluentAssertions 8.x` для читабельних assertions.

**Ключовий виклик:** FluentAssertions 8.x змінив API для nullable reference types — перехід на `Assert.NotNull()` / `Assert.IsType<T>()` де необхідно.

## Висновки

Проєкт Lumina реалізує систему управління університетом з чіткою архітектурою, реальними патернами проєктування та покриттям тестами. Clean Architecture забезпечує незалежність шарів і легке розширення. Сім патернів GoF вирішують конкретні архітектурні проблеми, а не є формальним доповненням.

Система готова до production-розгортання на SQLite (невеликий університет) або SQL Server (масштабоване середовище). Архітектура дозволяє додавати нові модулі (стипендії, міжнародний обмін, API) без переписування існуючого коду.
