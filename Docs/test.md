# Звіт про  тестування проекту UniversityManagement

У цьому документі описано структуру та покриття модульними тестами (Unit Tests) системи управління університетом (`UniversityManagement.Tests`).

##  Використані технології та інструменти
- **Фреймворк тестування:** `xUnit`
- **Бібліотека для тверджень (Assertions):** `FluentAssertions` (забезпечує зручний та читабельний синтаксис перевірок, наприклад `result.Should().BeTrue()`).
- **Ізоляція залежностей (Mocking):** `Moq` (для імітації роботи репозиторіїв).
- **База даних для тестів:** `Microsoft.EntityFrameworkCore.InMemory` (дозволяє тестувати контролери в пам'яті без реальної БД)


##  Що саме тестується? 

### 1. `AuthenticationServiceTests` (Тестування автентифікації та паролів)
Цей клас перевіряє правила валідації паролів за допомогою параметризованих тестів (`[Theory]` та `[InlineData]`):
- **Складність пароля:** Перевіряється, чи містить пароль мінімум 8 символів, великі та малі літери, цифри та спеціальні символи (наприклад, `Admin123!`).
- **Невалідні паролі:** Відхиляються паролі, які є занадто короткими, не мають великих/малих літер або цифр.
- **Обробка крайових випадків:**
  - Порожній пароль визначається як невалідний.
  - Паролі з пробілами розпізнаються і коректно обробляються.

### 2. `FactoryTests` (Тестування патерну Factory)
Перевіряється коректна робота фабричного методу для створення різних типів занять:
- **`LectureFactory`, `LaboratoryFactory`, `SeminarFactory`:** Кожна фабрика повинна створювати відповідний об'єкт уроку (`Lesson`) з правильним заголовком (Topic) та типом (Enum `LessonType`).
- **`LessonFactoryProvider`:** Перевіряється отримання правильної фабрики на основі рядкового вводу (наприклад, `"lecture"` повертає `LectureFactory`).
- **Унікальність типів:** Гарантується, що різні фабрики генерують уроки з унікальними типами.

### 3. `RoleAccessTests` (Тестування доступу ролей)
- Проста перевірка того, що рядок ролі `"Administrator"` ініціалізується та зберігає своє значення правильно.

### 4. `StudentsControllerTests` (Тестування веб-контролера MVC)
Цей клас тестує логіку обробки HTTP-запитів контролером `StudentsController` з використанням бази даних In-Memory та спеціально створеного `FakeStudentRepository`:
- **Метод `Index`:**
  - Повертає `ViewResult` зі списком студентів, якщо вони є в БД.
  - Повертає порожній `ViewResult`, якщо студентів немає.
- **Методи `Details` та `Edit`:**
  - Якщо запрошеного студента (за неіснуючим ID) немає в базі, контролер коректно повертає `NotFoundResult` (помилка 404).

### 5. `StudentServiceTests` (Тестування бізнес-логіки сервісу студентів)
Клас використовує `Moq` для створення "заглушки" (mock) `IRepository<Student>`, що дозволяє тестувати сервіс ізольовано від реальної БД:
- **Отримання даних (`GetAllAsync`, `GetByIdAsync`):**
  - Перевіряється повернення всіх студентів або порожнього списку.
  - Перевіряється пошук студента за ID (успішний пошук повертає студента, неуспішний — `null`).
- **Додавання та видалення (`AddStudentAsync`, `DeleteStudentAsync`):**
  - При додаванні нового студента перевіряється, чи був викликаний метод `AddAsync` у репозиторії рівно один раз (`Times.Once`).
  - При видаленні: якщо студент існує, метод `DeleteAsync` викликається. Якщо студента немає — метод видалення ігнорується (`Times.Never`).

### 6. `StudentTests` (Тестування моделі даних)
- Перевіряється правильність створення об'єкта `Student` (призначення імені, прізвища, email-адреси та рейтингу). Усі поля мають коректно зберігати призначені їм значення.
## Висновок
Тестове покриття охоплює всі ключові рівні архітектури додатку:
1. **Рівень даних (Models):** Перевірка створення сутностей (`StudentTests`).
2. **Рівень бізнес-логіки (Services/Patterns):** Валідація паролів, патерни створення об'єктів та бізнес-правила (`AuthenticationServiceTests`, `FactoryTests`, `StudentServiceTests`).
3. **Рівень презентації/API (Controllers):** Перевірка маршрутизації та відповідей HTTP (`StudentsControllerTests`).

Код написаний з дотриманням принципів AAA (Arrange - Act - Assert) та використовує сучасні підходи до мокінгу і перевірки результатів.
